## Graph Node Architecture in LangGraph

LangGraph employs a directed graph structure to orchestrate agent behavior, utilizing specialized nodes interconnected by edges. This architecture facilitates complex reasoning and action sequences. The core agent flow consists of eight distinct node types arranged in a predefined, yet dynamically routed, sequence: START → memory_node → router_node → (conditional: retrieval_node OR tool_node OR skip_node) → answer_node → eval_node → (conditional: retry answer_node OR save_node) → END.

**Node Specialization:** Each node encapsulates a specific function within the agent’s operation. `memory_node` manages conversational history. `router_node` determines the subsequent action based on the current state. `retrieval_node` interacts with external knowledge sources. `tool_node` executes defined tools (e.g., APIs, calculators). `skip_node` allows the agent to proceed without external interaction. `answer_node` generates a response based on accumulated information. `eval_node` assesses the quality of the answer. `save_node` persists relevant information.

**Edge Types: Fixed vs. Conditional:**  The graph’s connectivity is defined by two edge types. *Fixed edges*, established using `add_edge()`, represent deterministic transitions. For example, the START node always transitions to the memory_node. *Conditional edges*, created with `add_conditional_edges()`, introduce dynamic routing. These edges rely on decision functions that evaluate the agent’s state and determine the next node.

**Decision Functions & Routing:** The `router_decision` function is a critical component. It receives the agent’s state dictionary and inspects the `state['route']` key. Based on its value (typically 'retrieve', 'tool', or 'skip'), it directs the flow to the corresponding node. Similarly, the `eval_decision` function analyzes `state['faithfulness']` (a metric indicating answer accuracy) and `state['eval_retries']` (the number of evaluation attempts). If faithfulness is low and retry attempts are available, the agent loops back to the `answer_node` for refinement. Otherwise, it proceeds to the `save_node`.

**API Requirements & Testability:** LangGraph’s API mandates that decision functions are defined as standalone Python functions (callables) *outside* of the node’s internal `invoke()` method. This separation is crucial for several reasons. First, it enhances testability, allowing independent verification of routing and evaluation logic. Second, it adheres to LangGraph’s internal architecture, ensuring compatibility and proper execution within the graph framework.  The state dictionary passed to these functions provides a comprehensive view of the agent’s current context, enabling informed decision-making.
