## Retrieval-Augmented Generation (RAG) Pipeline: A Deep Dive

Retrieval-Augmented Generation (RAG) is a powerful paradigm for enhancing Large Language Model (LLM) performance by grounding responses in external, verifiable knowledge. This approach mitigates the inherent limitations of LLMs – namely, their potential for generating factually incorrect or outdated information (hallucination) – by providing them with relevant context *during* inference. The RAG pipeline implemented in this project consists of five core stages: Document Preparation, Embedding, Storage, Retrieval, and Generation.

**1. Document Preparation:** The initial step involves preparing domain-specific documents for processing. Raw documents are segmented into smaller chunks, typically ranging from 100 to 500 words. This chunking strategy is crucial for two reasons. First, it allows for more granular retrieval of relevant information. Second, and critically, it adheres to the token limits of the embedding model. Specifically, we utilize a chunk size designed to remain within the 256-token limit of the SentenceTransformer model used in the subsequent embedding stage. Overly large chunks can lead to information loss during embedding, while excessively small chunks may lack sufficient context for effective retrieval.

**2. Embedding:** Each text chunk is transformed into a dense vector representation using the `all-MiniLM-L6-v2` SentenceTransformer model. This model maps semantically similar text to nearby points in a high-dimensional vector space. The `all-MiniLM-L6-v2` model is chosen for its balance of performance and efficiency, offering strong semantic understanding with a relatively small model size. The resulting embedding is a numerical vector capturing the meaning of the chunk.

**3. Storage:** The generated embeddings, along with associated metadata (e.g., document source, chunk ID), are stored in a vector database – ChromaDB in this implementation. ChromaDB is optimized for similarity search, enabling efficient retrieval of embeddings closest to a given query embedding. Metadata is essential for provenance tracking and filtering retrieved results.

**4. Retrieval:** When a user submits a query, the query itself is first embedded using the same `all-MiniLM-L6-v2` SentenceTransformer model. This query embedding is then used to perform a similarity search within ChromaDB. The pipeline retrieves the *k* most similar embeddings (in this case, *k*=3), representing the top 3 most relevant text chunks. Similarity is typically measured using cosine similarity.

**5. Generation:** The retrieved context (the 3 most relevant text chunks) is injected into the system prompt of the LLM. The prompt is carefully crafted to instruct the LLM to *exclusively* base its answer on the provided context. This constraint is paramount for preventing hallucination and ensuring factual accuracy. The LLM then generates a response grounded in the retrieved information, effectively leveraging external knowledge to provide informed and reliable answers.
